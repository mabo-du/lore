# From Python Script to Zero-Dependency Executable: A PyInstaller Blueprint for Lore’s CPU-Optimized, Sub-400MB Transcription App

## Core Dependency Strategy: Balancing Performance, Size, and Startup Time

The foundational decision in architecting the Lore application for distribution is the selection of its core machine learning dependencies, specifically for the Voice Activity Detection (VAD) component. This choice directly impacts the two most stringent constraints of the project: the maximum installed bundle size of under 400 MB and the target cold startup time of under 5 seconds. The available options for `silero-VAD` present a critical trade-off between functionality, bundle footprint, and performance overhead. An analysis of the provided sources indicates that leveraging a specialized, lightweight library is the most strategic path forward, while the `faster-whisper` dependency presents a different set of challenges related to binary packaging and runtime behavior. The primary challenge for `faster-whisper` lies in its reliance on the CTranslate2 inference engine, which consists of native C++ libraries that PyInstaller cannot automatically detect [[9](https://pypi.org/project/faster-whisper/), [34](https://pypi.org/project/faster-whisper/0.3.0/)]. Unlike pure Python packages, these binaries must be explicitly included in the final executable. This involves identifying the specific shared library files (`.dll` on Windows, `.so` on Linux, `.dylib` on macOS) distributed with the `ctranslate2` package and instructing PyInstaller to copy them into the output directory. While specific `hiddenimports` may be necessary for Python modules that interface with this engine, the more critical task is managing the underlying native binaries. Furthermore, the user's requirement for a CPU-only deployment is paramount. It is essential to select wheels for `faster-whisper` and `CTranslate2` that do not contain optional GPU-specific code. On Windows, this means ensuring no `cudatoolkit` dependencies or associated DLLs are bundled. If such files are inadvertently included, they can be manually excluded via the `binaries` list in the PyInstaller `.spec` file, but preventing their inclusion at the installation stage is preferable. The presence of CUDA libraries in a CPU-only build can lead to silent failures or significant delays as the application attempts to initialize a non-existent GPU runtime [[27](https://stackoverflow.com/questions/79765261/how-to-handle-could-not-initialize-nnpack-reason-unsupported-hardware-warnin)].

The decision for `silero-VAD` is even more acute due to the project's aggressive size constraint. The evidence strongly suggests that using PyTorch for this purpose is untenable. While PyTorch offers flexibility, bundling its CPU-only wheel is known to significantly increase the final executable size, likely pushing the bundle past the 400 MB limit even before accounting for other dependencies like `PyQt6` and `ffmpeg` [[27](https://stackoverflow.com/questions/79765261/how-to-handle-could-not-initialize-nnpack-reason-unsupported-hardware-warnin), [29](https://www.zhihu.com/en/answer/3132384280)]. There are also documented instances where PyTorch models fail to load within a PyInstaller-packaged environment, sometimes due to initialization errors related to CPU instruction sets like NNPACK on certain processors [[27](https://stackoverflow.com/questions/79765261/how-to-handle-could-not-initialize-nnpack-reason-unsupported-hardware-warnin), [29](https://www.zhihu.com/en/answer/3132384280)]. These issues represent a high risk to project stability. In contrast, using ONNX Runtime is a more optimized approach for inference tasks [[22](https://pypi.org/project/onnxruntime/)]. ONNX Runtime is designed as a high-performance scoring engine for pre-trained models, allowing for extensive graph-level optimizations that are difficult to achieve with PyTorch's dynamic computation graph [[45](https://stackoverflow.com/questions/67943173/onnxruntime-vs-pytorch)]. However, the standard `onnxruntime` package itself still carries a notable dependency footprint. The most compelling solution identified is the use of `silero-vad-lite` [[26](https://pypi.org/project/silero-vad-lite/0.1.0/)]. This package is explicitly designed to provide a simple interface to the Silero VAD model without requiring the full `onnxruntime` Python package. This makes it an ideal candidate for meeting the project's strict size requirements. By choosing `silero-vad-lite`, the development team can eliminate the dependency on both PyTorch and the larger ONNX Runtime package, thereby minimizing the bundle size and reducing the complexity of the dependency chain. This choice aligns perfectly with the goal of creating a minimal, fast, and reliable application. The performance difference between `silero-vad-lite` and a full ONNX implementation is expected to be negligible for the use case, as the primary computational workload remains the same; the main benefit is the reduction in packaging overhead and potential startup latency associated with importing a large framework like PyTorch or ONNX Runtime. Therefore, the strategic recommendation is to adopt `silero-vad-lite` as the VAD backend, a decision that directly addresses the project's most challenging constraints.

| Dependency | Recommended Backend | Rationale |
| :--- | :--- | :--- |
| `faster-whisper` / `CTranslate2` | CPU-only wheel | Ensures compatibility with standard office hardware by excluding CUDA dependencies, which could cause silent failures or startup delays [[27](https://stackoverflow.com/questions/79765261/how-to-handle-could-not-initialize-nnpack-reason-unsupported-hardware-warnin)]. |
| `silero-VAD` | `silero-vad-lite` | Minimizes bundle size by avoiding the large `torch` and `onnxruntime` dependencies, which is critical for staying under the 400 MB constraint [[26](https://pypi.org/project/silero-vad-lite/0.1.0/)]. |
| `ffmpeg` | Bundled via `imageio-ffmpeg` | Ensures a zero-dependency user experience by including the necessary binary within the application bundle, simplifying distribution [[63](https://toolbelt.readthedocs.io/en/latest/downloadutils.html)]. |

This strategic foundation—prioritizing a lean, CPU-optimized dependency stack—is the first and most crucial step toward achieving the desired outcome for the Lore application.

## PyInstaller Configuration and Cross-Platform Packaging

Achieving a stable, cross-platform Lore application with PyInstaller requires meticulous configuration of the `.spec` file and a deep understanding of the idiosyncrasies of its target dependencies. The primary challenges revolve around Qt plugin discovery, lazy-loading large modules to maintain responsiveness, and navigating platform-specific gotchas. For PyQt6, one of the most common and critical errors is the failure to find the Qt platform plugin, resulting in messages like "Could not find or load the Qt platform plugin 'windows'" on Windows, or equivalent errors on macOS and Linux [[70](https://askubuntu.com/questions/1485442/issue-with-installing-pyqt6-on-ubuntu-22-04), [71](https://forum.qt.io/topic/122641/qt-qpa-plugin-could-not-find-the-qt-platform-plugin-windows-in)]. This issue arises because PyInstaller's analysis does not always include the entire `qt6_plugins` directory from the PyQt6 installation. The definitive solution is to explicitly add this directory to the `Tree` object within the `.spec` file, ensuring its contents are copied into the final bundle. This single directive is often sufficient to resolve startup crashes on all major platforms. Beyond plugin handling, UI freezing is a significant concern, especially during long-running operations like model downloading [[65](https://forum.qt.io/topic/139550/ui-freezing-phenomenon-in-macos-pyside6), [66](https://zhuanlan.zhihu.com/p/655541311)]. Any operation that blocks the main event loop will render the application unresponsive. While the proposed first-run wizard will block the main window until the download completes, it is still imperative to execute the download logic in a background thread. Using PyQt's threading mechanisms, such as `QThread` or `QThreadPool` with `QRunnable`, is the correct approach to prevent the entire GUI from freezing [[18](https://stackoverflow.com/questions/6783194/background-thread-with-qthread-in-pyqt), [19](https://realpython.com/python-pyqt-qthread/)]. This ensures that if any minor UI updates or events need processing during the wait, they are handled smoothly. Although the main window is modal, background threading is a defensive programming practice that enhances robustness and user experience.

While alternatives like BeeWare's `Briefcase` exist and are promoted by the Qt documentation as a tool for freezing applications [[15](https://doc.qt.io/qtforpython-6/deployment/deployment-pyinstaller.html)], PyInstaller remains the superior choice for this specific project. `Briefcase` introduces another layer of abstraction and may not offer the fine-grained control necessary to troubleshoot and resolve the complex interactions between `faster-whisper`, `PyQt6`, and the various system libraries. PyInstaller's maturity, extensive community support, and detailed hook system make it better suited for debugging and customizing the packaging process for this unique combination of dependencies. Similarly, `cx_Freeze` is another viable option but has historically shown less proficiency in handling complex binary dependencies compared to PyInstaller [[68](https://askubuntu.com/questions/1465218/pip-error-on-ubuntu-externally-managed-environment-%C3%97-this-environment-is-extern)]. Therefore, committing to PyInstaller is the most pragmatic path. The following table outlines the essential `hiddenimports`, `datas`, and `binaries` required for a functional Lore application, synthesized from the analysis of the provided sources.

This configuration serves as a robust starting point for the `lore.spec` file. It proactively addresses the most common pitfalls encountered when packaging AI-driven Qt applications, laying the groundwork for a stable and portable distribution.

## First-Run Model Management and Storage Architecture

Designing a resilient and user-friendly first-run experience is critical for the success of Lore, particularly given that it involves downloading large machine learning models. The user's requirement for explicit, upfront model selection is a sound design choice, as it educates the user about the trade-offs between transcription speed and disk space usage—a crucial consideration for a non-technical audience [[38](https://arxiv.org/html/2507.14451v1), [52](https://voice-mode.readthedocs.io/en/latest/guides/whisper-setup/)]. The optimal UX pattern involves presenting a dedicated setup wizard on first launch. This wizard should feature a simple, intuitive interface, such as a dialog with radio buttons labeled "Fast," "Balanced," and "Best Quality." Each option must be accompanied by clear size disclosures, for example: "Small Model (~750MB)" or "Large Model (~1.5GB)," referencing the sizes of available Whisper models [[39](https://pypi.org/project/openai-whisper/)]. This transparency empowers the user to make an informed decision based on their hardware and workflow needs. Once a model is selected, the application should begin the download process immediately. To maintain a responsive UI during this potentially lengthy operation, the download must be executed in a background thread using a mechanism like `QThread` or `QThreadPool` [[18](https://stackoverflow.com/questions/6783194/background-thread-with-qthread-in-pyqt), [19](https://realpython.com/python-pyqt-qthread/)]. Even though the main application window will be blocked behind the modal wizard, background threading prevents the entire application from freezing and allows for smooth handling of any internal events. The wizard should display a clear progress bar indicating the download progress, providing immediate visual feedback to the user. Upon successful completion, the wizard would transition to a "Ready" state, and the main application window would become accessible.

A cornerstone of a professional-grade application is reliability, and this extends to the download process itself. Interruptions due to network instability are a common occurrence. To address this, the download manager must implement resumable downloads. Modern HTTP servers support this capability through the `Range` header [[2](https://www.oreilly.com/library/view/python-cookbook/0596001673/ch11s06.html)]. When a download is initiated, the client should first check if a partial file already exists. If so, it can send a subsequent request with a `Range: bytes=<start>-` header to continue the download from the last known byte position, rather than restarting from the beginning [[6](https://devpress.csdn.net/python/62fd9ff77e66823466192b12.html), [31](https://stackoverflow.com/questions/65584011/how-to-reconnect-in-requests-to-continue-a-download)]. This feature is a significant quality-of-life improvement, saving users considerable time and bandwidth. The `requests-toolbelt` library provides convenient utilities for streaming responses and monitoring download progress, which can simplify the implementation of both the progress bar and the resumable logic [[62](https://blog.csdn.net/summerpowerz/article/details/80293235), [63](https://toolbelt.readthedocs.io/en/latest/downloadutils.html)]. Regarding where to store these downloaded models, hard-coding platform-specific paths is poor practice. The `platformdirs` library is the definitive solution for this problem, offering a standardized way to locate appropriate directories across all operating systems [[10](https://platformdirs.readthedocs.io/en/latest/api.html), [11](https://pypi.org/project/platformdirs/3.10.0/)]. The application should use `platformdirs.user_data_dir('Lore', 'YourAppAuthor')` to get the base directory for user-specific data. Within this directory, a dedicated `models` subdirectory should be created to store all downloaded model files. This approach ensures that user data is stored in the conventionally expected location on each platform:
*   **Windows:** `%APPDATA%\\Lore\\models` (typically `C:\Users\<User>\AppData\Local\Lore\models`)
*   **macOS:** `~/Library/Application Support/Lore/models`
*   **Linux:** `~/.local/share/Lore/models`

This adherence to OS conventions is crucial for a seamless user experience. Finally, the question of whether to check for model updates on subsequent launches is important. For a local model cache, a "download once, never update" policy is acceptable and even preferable from a simplicity perspective. Whisper models are large, and automatic background updates could consume significant bandwidth without the user's explicit consent. The application could, however, display a notification on launch if a new version of the model is available, allowing the user to choose when and if they wish to update. This balances the desire for up-to-date models with respect for the user's time and data plan.

## CI/CD Pipeline and Final Build Optimization

To ensure consistent and reliable builds across all target platforms, establishing a robust CI/CD pipeline is essential. GitHub Actions is the ideal platform for this task, offering pre-configured virtual environments for Windows, macOS, and Linux that can be leveraged to create parallel build jobs [[68](https://askubuntu.com/questions/1465218/pip-error-on-ubuntu-externally-managed-environment-%C3%97-this-environment-is-extern)]. A single workflow file can define three distinct jobs, one for each operating system. The Windows job would run on the `runs-on: windows-latest` runner, targeting Windows 10+ compatibility. The macOS job should utilize the `runs-on: macos-latest` runner, but to maximize performance on modern Apple Silicon Macs, it is highly recommended to specify the ARM architecture (`permissions: {}`). This ensures the application is built natively for `arm64`, avoiding performance penalties from Rosetta translation [[14](https://stackoverflow.com/questions/76866923/pyqt6-pyinstaller-app-builds-but-wont-open-macos-m1-chip), [67](https://juejin.cn/post/7107498281319530509)]. The Linux job would run on `ubuntu-22.04` or `ubuntu-latest`, providing a stable environment for building against glibc 2.35 or newer. This parallel execution model dramatically reduces the total time required to produce all three artifacts. The workflow should encapsulate the entire build process: setting up the Python environment, installing dependencies from a lockfile (e.g., `requirements.txt`), running PyInstaller with the finalized `.spec` file, and finally, performing code signing and artifact compression. Integrating these final steps is critical for producing a professional-quality distribution package.

Code signing is not merely an optional extra; it is a mandatory step for professional software distribution. An unsigned executable on Windows will trigger SmartScreen warnings, which can deter users from running the application [[57](https://stackoverflow.com/questions/43777106/program-made-with-pyinstaller-now-seen-as-a-trojan-horse-by-avg)]. On macOS, an unsigned application will fail to open on recent versions like Ventura and later without the user manually overriding security settings, and it will not be notarized [[75](https://cloud.tencent.cn/developer/information/%E4%B8%BA%E4%BB%80%E4%B9%88macOS%E4%B8%8A%E7%9A%84PyQt%E8%8F%9C%E5%8D%95%E4%BC%9A%E5%86%BB%E7%BB%93%EF%BC%9F)]. Notarization is a process where Apple scans the app for malicious components, and without a valid developer certificate and successful notarization, Gatekeeper will block its execution. Therefore, acquiring the necessary certificates for both Windows and macOS and automating the signing process within the GitHub Actions workflow is non-negotiable for a credible release.

Finally, optimization techniques like UPX compression should be employed. UPX is a free, open-source executable packer that can significantly reduce the size of the final executables without altering their functionality. Given the tight 400 MB bundle size constraint, every megabyte counts. Enabling UPX compression via the `--upx-dir` flag in the PyInstaller command is a standard practice and should be enabled for all builds. There are no known conflicts between UPX and the specified dependencies, including `faster-whisper` or `PyQt6`. The empirical results of the build—final bundle size and measured startup time—are the ultimate arbiters of success. While an initial estimate of 300-350 MB seems achievable with the chosen lean dependencies, the actual figures must be verified through testing. The startup time is heavily dependent on the benchmark hardware, but employing strategies like lazy-loading non-critical modules can help meet the sub-5-second target [[83](https://pypi.org/project/pyinstaller/3.3/)]. The following table summarizes the recommended build configurations.

| Platform | GitHub Actions Runner | Minimum OS Version | Key Build Considerations |
| :--- | :--- | :--- | :--- |
| **Windows** | `windows-latest` | Windows 10 x64 | Code signing with a trusted certificate is required to avoid SmartScreen warnings. Ensure CTranslate2 MSVC runtime is compatible. |
| **macOS** | `macos-latest` (ARM) | macOS Ventura | Native ARM (arm64) build is strongly recommended for performance. Code signing and notarization are mandatory. |
| **Linux** | `ubuntu-22.04` | Ubuntu 22.04+ | No code signing required. Target glibc 2.35+. Bundle `ffmpeg` using `imageio-ffmpeg`. |

By implementing this comprehensive CI/CD strategy, the development team can automate the creation of fully signed, compressed, and professionally packaged artifacts for all platforms, ensuring a high-quality and secure user experience.

## Synthesis and Strategic Recommendations

This research provides a comprehensive technical blueprint for the successful PyInstaller-based packaging of the Lore desktop application. The primary objective is to deliver a robust, cross-platform tool for non-technical users that operates reliably within strict constraints on size (<400 MB) and startup time (<5 seconds). The analysis reveals that achieving this goal hinges on three strategic pillars: a deliberate choice of lean, CPU-optimized dependencies; meticulous and proactive management of PyInstaller's packaging directives; and the implementation of a user-centric, resilient first-run experience.

First, the core dependency strategy must prioritize minimalism. The most critical decision was the selection of `silero-vad-lite` over both PyTorch and the full ONNX Runtime. This choice is the single most effective measure for keeping the application bundle well within the 400 MB size limit, as it eliminates the substantial overhead of larger ML frameworks [[26](https://pypi.org/project/silero-vad-lite/0.1.0/)]. For `faster-whisper`, the focus must be on securing a CPU-only wheel and explicitly ensuring that no CUDA libraries are bundled, which is essential for stability on standard office hardware [[27](https://stackoverflow.com/questions/79765261/how-to-handle-could-not-initialize-nnpack-reason-unsupported-hardware-warnin)].

Second, PyInstaller configuration must be granular and precise. A generic build will fail. The `.spec` file must be customized to explicitly include the `qt6_plugins` directory to prevent platform-specific startup failures [[54](https://stackoverflow.com/questions/47468705/pyinstaller-could-not-find-or-load-the-qt-platform-plugin-windows), [71](https://forum.qt.io/topic/122641/qt-qpa-plugin-could-not-find-the-qt-platform-plugin-windows-in)]. Furthermore, the application's architecture must incorporate background threading for all long-running I/O operations, such as the first-run model download, to maintain a responsive UI and prevent application freezes [[18](https://stackoverflow.com/questions/6783194/background-thread-with-qthread-in-pyqt), [19](https://realpython.com/python-pyqt-qthread/)]. Adhering to best practices for cross-platform data storage using the `platformdirs` library is also fundamental for a seamless user experience across Windows, macOS, and Linux [[10](https://platformdirs.readthedocs.io/en/latest/api.html), [11](https://pypi.org/project/platformdirs/3.10.0/)].

Third, the first-run model download flow must be designed for clarity and resilience. The UX should feature explicit model selection with clear size and performance trade-offs disclosed to the user. Under the hood, this process must leverage a resumable download mechanism using HTTP Range requests to handle interruptions gracefully, a hallmark of professional software [[2](https://www.oreilly.com/library/view/python-cookbook/0596001673/ch11s06.html), [31](https://stackoverflow.com/questions/65584011/how-to-reconnect-in-requests-to-continue-a-download)]. All downloaded models must be stored in a dedicated, OS-appropriate location managed by `platformdirs`.

Finally, the entire build and distribution process should be automated using a parallelized GitHub Actions workflow. This workflow must perform PyInstaller builds for all three platforms, followed by mandatory code signing and notarization to ensure trust and compliance with platform security policies [[57](https://stackoverflow.com/questions/43777106/program-made-with-pyinstaller-now-seen-as-a-trojan-horse-by-avg), [75](https://cloud.tencent.cn/developer/information/%E4%B8%BA%E4%BB%80%E4%B9%88macOS%E4%B8%8A%E7%9A%84PyQt%E8%8F%9C%E5%8D%95%E4%BC%9A%E5%86%BB%E7%BB%93%EF%BC%9F)]. UPX compression should be enabled to further optimize the final bundle size.

In conclusion, the successful packaging of Lore is entirely feasible with PyInstaller. It requires moving beyond a simple `pyinstaller --onefile main.py` command and embracing a more sophisticated, targeted approach. By following the recommendations outlined in this report—including the adoption of `silero-vad-lite`, careful `.spec` file configuration, robust background threading, and a resilient download manager—the development team can confidently produce a high-quality, zero-dependency executable that meets all the specified goals and delivers an excellent experience to its target users.
