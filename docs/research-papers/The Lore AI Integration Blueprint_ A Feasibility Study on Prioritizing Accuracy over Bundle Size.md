# The Lore AI Integration Blueprint: A Feasibility Study on Prioritizing Accuracy over Bundle Size

This report presents a comprehensive technical feasibility analysis for integrating three advanced offline AI capabilities—Named Entity Recognition (NER), audio-signal non-verbal event detection, and local Large Language Model (LLM) integration—into the Lore desktop application. The evaluation rigorously adheres to the project's strict architectural constraints: no PyTorch dependencies, a sub-400 MB baseline bundle size with approximately 30 MB of remaining headroom, CPU-only execution, and defined RAM budgets. For NER, the analysis prioritizes accuracy over bundling, mandating that all models be treated as optional post-install downloads. Each section provides a detailed assessment across five key dimensions—Technical Feasibility, UX Friction, Bundle & Disk Impact, Runtime Resources, and Maintenance & Licensing—and concludes with a concrete recommended stack, bundle delta, and PyQt6 integration sketch. The final synthesis offers a strategic recommendation for feature priority, cumulative overhead estimation, phased implementation, and a unified pipeline architecture.

## Named Entity Recognition: Achieving High Accuracy via Optional Downloads

The primary goal of implementing Named Entity Recognition (NER) is to automatically identify and tag mentions of people, places, and organizations within the transcript generated by faster-whisper. This functionality would provide significant semantic enrichment for oral history archivists, surfacing entities as inline annotations and filterable sidebar entries. The critical architectural constraint is the explicit mandate to prioritize accuracy over bundling, which dictates that all NER models must be optional downloads rather than being included in the ~370 MB baseline application bundle. This immediately disqualifies any model whose combined library and model weight would exceed the available headroom, shifting the focus toward high-performance, downloadable solutions. The core challenge lies in finding a PyTorch-free inference path for a model powerful enough to achieve high accuracy on the informal, domain-specific language of conversational oral history.

From a technical feasibility perspective, several options exist that comply with the "no PyTorch" rule. The standard spaCy English pipelines, `en_core_web_sm` and `en_core_web_md`, are strong candidates due to their pure-Python/C++ backend and lack of a runtime PyTorch dependency [[7](https://pypi.org/project/gliner/0.2.5/)]. These models are lightweight and suitable for bundling, but their accuracy may be limited. The `sm` variant uses statistical models, offering fast performance at the cost of nuanced understanding, while `md` incorporates word vectors, providing better accuracy at the expense of larger disk and RAM footprint [[7](https://pypi.org/project/gliner/0.2.5/)]. Quantifying the accuracy gain of `md` over `sm` on the specific domain of oral history is crucial; however, even `md` is likely to struggle with the non-standard proper nouns, regionalisms, and informal grammar common in spoken interviews. Another option is GLiNER, a generalist NER model capable of identifying any entity type specified at runtime using a bidirectional transformer encoder [[7](https://pypi.org/project/gliner/0.2.5/), [9](https://www.kaggle.com/datasets/dschettler8845/gliner-github-repo), [10](https://arxiv.org/html/2401.10825v2)]. Its zero-shot capability is particularly well-suited for oral history, where entities like historical periods ("the war"), Indigenous place names, or family surnames used informally are not present in standard training corpora [[8](https://modelscope.cn/models/knowledgator/gliner-pii-edge-v1.0)]. The major hurdle for GLiNER is its reliance on PyTorch for its native implementation. However, exporting a trained PyTorch model to ONNX is a well-established process [[16](https://docs.pytorch.org/tutorials/beginner/onnx/export_simple_model_to_onnx_tutorial.html), [21](https://www.aidoczh.com/torchtutorials/advanced/super_resolution_with_onnxruntime.html)]. The viability of this approach depends entirely on the existence of a stable, community-maintained ONNX export of a GLiNER model. If such an export exists and performs reliably on CPU, it represents the most promising path to achieving the desired accuracy. Finally, libraries designed from the ground up for ONNX-based inference, such as `edsnlp`, could offer a production-grade alternative, though they may have a smaller ecosystem compared to spaCy or GLiNER [[15](https://arxiv.org/html/2602.18487v1)].

The user experience for NER must be seamless for non-technical archivists. The workflow would involve a simple button click after transcription completes to initiate the NER process. The primary UX friction point arises from the interaction between NER and the existing faster-whisper configuration. Since `condition_on_previous_text=False` causes each 30-second segment to be transcribed independently, consistent spelling of proper nouns across chunk boundaries can be lost [[13](https://www.datalearner.com/benchmark-compare/Phi-4-mini-instruct/Llama-3_2-3B/Qwen2_5-3B/Qwen2_5-7B/Llama3_1-8B)]. An archivist might see "Smith" in one segment and "Smyth" in another, leading to fragmented entity recognition. To mitigate this, a post-processing aggregation step is essential. All entity mentions would be collected from the full transcript, normalized using fuzzy matching or lemmatization algorithms, and then the entire transcript would be re-scanned to apply consistent, canonical labels. This adds computational complexity but is necessary for reliable results. Integrating the results into the Lore UI requires extending the existing `QTextBlockUserData` structure. The `SegmentData` object already stores timing and speaker information per block. NER entities could be stored as a list of dictionaries (e.g., `{'text': 'John Smith', 'label': 'PERSON'}`) within this object. When rendering the `QPlainTextEdit`, a custom `QSyntaxHighlighter` or direct application of `QTextCharFormat` overlays can be used to visually highlight these entities in the editor. This approach is performant even with thousands of blocks, as formatting changes are typically applied only to the visible viewport [[1](https://qwenlm.github.io/blog/qwen2.5/)].

The integration into Lore's threading model is straightforward. A new `QThread` worker, `NERWorker`, would be created. Upon receiving the complete transcript text, it would load the appropriate NER model (either from the bundled set or a downloaded location) and process the text. Inference latency on a 30-second segment needs to be evaluated, but on modern CPUs, even transformer-based models can process text quickly. Once processing is complete, the worker emits a signal containing the list of detected entities with their corresponding timestamps. The main thread receives this signal and iterates through the entities, updating the `QTextBlockUserData` for the relevant blocks and triggering a repaint. This keeps the main UI thread free and responsive.

[[7](https://pypi.org/project/gliner/0.2.5/), [8](https://modelscope.cn/models/knowledgator/gliner-pii-edge-v1.0), [16](https://docs.pytorch.org/tutorials/beginner/onnx/export_simple_model_to_onnx_tutorial.html)] |

Based on this analysis, the recommended stack for NER is **GLiNER via a community-maintained ONNX export**. This choice directly addresses the user's primary requirement for high accuracy on domain-specific text. The bundle delta is not applicable as the model is an *optional download*. The integration sketch involves creating a `NERWorker` `QThread` that processes the full transcript, normalizes entities to resolve chunk-boundary inconsistencies, and signals the main thread to update `QTextBlockUserData` with highlighted entities.

## Audio-Signal Non-Verbal Event Detection: Leveraging Existing VAD Outputs

The objective of non-verbal event detection is to automatically identify and timestamp events like `[Laughter]`, `[Crying]`, `[Silence]`, and `[Crosstalk]` directly from the audio signal, injecting these markers into the transcript. The guiding principle for this feature is to minimize added overhead by maximizing the reuse of existing infrastructure before introducing new models. The `silero-vad-lite` model, which is already running via `vad_filter=True`, provides a rich source of metadata that can be exploited for this purpose. Specifically, its output of `speech_timestamps` can be analyzed to detect gaps, which can be interpreted as potential silence events.

The most immediate and lowest-friction implementation is for `[Silence]` detection. The `silero-vad-lite` model produces a list of start and end times for speech segments [[3](https://www.kaggle.com/models/google/yamnet)]. By iterating through these timestamps, the application can calculate the duration of every non-speech gap. A simple algorithm can then classify gaps longer than a certain threshold (e.g., 3 seconds) as `[Silence: Xs]` annotations. This approach requires no additional model, incurs negligible computational cost, and adds virtually no overhead to the bundle size. It directly leverages the VAD output that is already being produced for transcription filtering. This creates a solid foundation for the feature with zero initial cost.

For more complex events like `[Laughter]` and `[Crying]`, which require audio event classification beyond simple energy thresholding, the analysis must consider small, efficient models that operate without PyTorch. YAMNet, a TensorFlow-based audio event classifier, is a prime candidate [[3](https://www.kaggle.com/models/google/yamnet)]. Its `.tflite` model format is designed for efficient deployment on resource-constrained devices, and the associated `tflite-runtime` package is very small (~2-3 MB) [[4](https://www.mathworks.com/help/audio/ug/audio-event-classification-using-tensorflow-lite-on-raspberry-pi.html), [5](https://stackoverflow.com/questions/76517158/i-need-help-for-using-yamnet-tensorflow-lite-model-tflite-for-sound-classific)]. The question is whether YAMNet's 521-class ontology includes relevant categories like laughter and crying [[3](https://www.kaggle.com/models/google/yamnet)]. Assuming it does, the per-30-second-window inference latency on a modern CPU would be extremely low, likely under 100ms. The total bundle impact would be the YAMNet model file itself, which is compact. An alternative is PANNs (Convolutional Audio Neural Networks), which are also known for high-quality audio classification [[20](https://pmc.ncbi.nlm.nih.gov/articles/PMC10347208/)]. MobileNetV2 variants of PANNs offer a good balance of efficiency and accuracy and could be exported to ONNX, allowing them to share the same inference runtime as `silero-vad-lite`. However, there is less public information about the availability of pre-trained, fine-tuned PANN models for these specific classes compared to YAMNet. Another speculative path is to analyze the raw probability output from `silero-vad-lite` for characteristic burst patterns that might correlate with laughter (short, rapid, high-confidence bursts). However, this is unreliable as VAD models are optimized for speech/non-speech distinction, not for recognizing complex audio events, and their probability scores may not be robust enough for this task.

Architecturally, the parallel pipeline should consist of a sibling `QThread` to the `TranscriptionWorker`, named `AudioClassifierWorker`. Both workers would operate on the same 16 kHz mono WAV buffer produced by ffmpeg. To avoid costly data copying between threads, this buffer should be shared using a `memoryview` or a NumPy array with a shared memory backing. The `AudioClassifierWorker` would process the audio in the same 30-second chunks as the transcription worker. As it identifies events, it would emit a signal containing a timestamped event label (e.g., `{'start_ms': 15000, 'end_ms': 15800, 'label': '[Laughter]'}'). On the main thread, a merge function would take the stream of transcription segments and the stream of event annotations, associating each event with the correct 30-second segment based on its timestamp. When both transcription and classification produce output for the same time window, a conflict resolution strategy is needed. A simple heuristic could be applied: if the transcription confidence is high, display the transcribed text; if the audio event is strong and the VAD indicates non-speech, insert the event token. The archivist must have the ability to easily review, accept, or reject each auto-inserted annotation directly in the transcript editor.

Given the nature of archival audio, which often suffers from low Signal-to-Noise Ratio (SNR), frequency roll-off, and tape hiss, the false positive rate for classifiers is a major concern [[6](https://arxiv.org/pdf/2305.12712)]. It is highly probable that a classifier trained on clean audio will generate many false positives for `[Crying]` and `[Laughter]` when faced with low-SNR recordings. Therefore, a mandatory archivist review step before these annotations are committed to the permanent transcript is not just recommended—it is essential for maintaining data integrity. The OHMS XML 6.0 specification, which uses a WebVTT-formatted transcript, does not have a standard syntax for non-verbal cues. A pragmatic solution is to represent these events as inline text tokens within the WebVTT cue text, for example: `WEBVTT ... 00:01:20.000 --> 00:01:25.000 [Laughter] That's just hilarious!`. The OHMS Viewer would need to be configured to render these characters visibly to distinguish them from spoken content.

[[3](https://www.kaggle.com/models/google/yamnet), [4](https://www.mathworks.com/help/audio/ug/audio-event-classification-using-tensorflow-lite-on-raspberry-pi.html)] |

The recommended stack is to **first implement `[Silence]` detection using the existing VAD gaps, and second, add YAMNet via `tflite-runtime` for `[Laughter]` and `[Crying]` if the accuracy proves insufficient.** The bundle delta for the base implementation is ~0 MB; adding YAMNet would add ~5-10 MB. The integration sketch involves a `QThread` sibling to `TranscriptionWorker` that shares the audio buffer and merges its timestamped event output with the transcription segment stream on the main thread.

## Local LLM Integration for Metadata Generation: Balancing Speed and Quality

The integration of a local LLM aims to automate the creation of a rich metadata suite for an oral history interview, including a plain-language abstract, thematic index points, keywords, and subject headings, all formatted for the OHMS XML 6.0 standard. The primary constraints are stringent: the operation must complete in under 5 minutes for a 60-minute interview and under 10 minutes for a 2-hour interview on a mid-range CPU, and the model must fit within a 5.7 GB RAM ceiling after the Whisper model is unloaded [[1](https://qwenlm.github.io/blog/qwen2.5/)]. This effectively caps the choice to quantized, smaller-scale models, likely in the 3-billion parameter range.

Several 3B-class models are strong candidates, all of which can be run using the `llama.cpp` runtime, ensuring a PyTorch-free environment. **Llama 3.2 3B** serves as a solid, well-rounded baseline with strong English reasoning abilities [[13](https://www.datalearner.com/benchmark-compare/Phi-4-mini-instruct/Llama-3_2-3B/Qwen2_5-3B/Qwen2_5-7B/Llama3_1-8B)]. **Phi-3.5 Mini 3.8B**, developed by Microsoft, is noted for its superior instruction-following and reasoning capabilities, which are critical for correctly generating structured JSON that maps to the complex OHMS schema [[13](https://www.datalearner.com/benchmark-compare/Phi-4-mini-instruct/Llama-3_2-3B/Qwen2_5-3B/Qwen2_5-7B/Llama3_1-8B)]. **Qwen 2.5 3B** stands out for its multilingual support across over 29 languages, making it a potentially valuable tool for generating English-language metadata from non-English transcripts [[1](https://qwenlm.github.io/blog/qwen2.5/)]. The RAM usage for these models, when quantized to the Q4_K_M level, is approximately 4.5-5.0 GB, comfortably below the 5.7 GB ceiling. The primary differentiator will be inference speed and quality on the specific task of structured metadata generation. On a modern Intel i7 CPU, all three models would be expected to generate between 10-20 tokens per second. On Apple M-series hardware, this could be significantly higher due to better CPU architecture and potential acceleration from BLAS optimizations within `llama.cpp` [[24](https://www.modelscope.cn/models/zt1224179632/llama.cpp/summary)].

A critical challenge stems from the established faster-whisper configuration of `condition_on_previous_text=False`, which results in independent transcriptions for each 30-second chunk. This can lead to inconsistent spellings and broken contextual flow at chunk boundaries. For LLM summarization, a **sliding window with overlap** strategy is superior to a map-reduce approach. Map-reduce would summarize sections independently and then try to synthesize the summaries, which can lose coherence. A sliding window, where subsequent chunks overlap by a few seconds (e.g., 5 seconds), allows the LLM to see context across the problematic boundaries, mitigating the effects of inconsistent proper noun spellings and preserving theme continuity. For a 60-minute interview yielding approximately 10,000 words (~13,000 tokens), a 3B model's context window is sufficient, especially with an efficient chunking strategy that minimizes redundant token throughput.

To guarantee the LLM produces usable output, `llama.cpp`'s support for Grammar-Based Normalised Form (GBNF) is a game-changing feature. Instead of relying on fragile prompt engineering to force JSON output, a GBNF grammar can be defined to strictly constrain the model's output to the exact schema required for OHMS XML generation [[17](https://hermes-agent.nousresearch.com/docs/zh-Hans/user-guide/skills/bundled/mlops/mlops-inference-llama-cpp)]. This eliminates the possibility of malformed JSON or hallucinated fields. The system prompt should be crafted from the persona of a professional archivist following OHA Best Practices, instructing the model to read the transcript, identify overarching themes, write a concise abstract, extract relevant keywords and LCSH-style subjects, and assign timestamps from the word-level timings provided by Whisper [[13](https://www.datalearner.com/benchmark-compare/Phi-4-mini-instruct/Llama-3_2-3B/Qwen2_5-3B/Qwen2_5-7B/Llama3_1-8B)]. This combination of a precise grammar and a well-defined persona prompt ensures high-fidelity, programmatic output.

The LLM's lifecycle must be managed carefully to adhere to the RAM budget. It should be loaded lazily, only when the user clicks the "Generate Abstract" button. The `llama_cpp.Llama` instance should be created within a dedicated `QThread` worker. The instantiation call would specify the path to the downloaded GGUF model file (from the `platformdirs` cache), set the context size (`n_ctx`), and explicitly disable GPU layers (`n_gpu_layers=0`) [[17](https://hermes-agent.nousresearch.com/docs/zh-Hans/user-guide/skills/bundled/mlops/mlops-inference-llama-cpp)]. After generation is complete, the worker should execute `del model` followed by `gc.collect()` to ensure the CTypes memory is freed. While `del` + `gc.collect()` is generally reliable for freeing the main model object, the underlying C library may hold onto some memory. The initial model load time from an NVMe SSD will be a few seconds, but from a slower spinning HDD, it could be ten or twenty seconds, which is a minor UX consideration. During token generation, streaming the output back to the main thread via a `pyqtSignal(str)` emitted for each new token provides a responsive, real-time feedback loop to the user, showing the LLM "thinking."

Offering the LLM download to a non-technical user requires careful UX design. Presenting it during the Phase 0 first-run wizard, alongside Whisper model selection, is the best approach. The options can be framed simply: "Fastest Transcription," "Balanced Quality," and "Best Quality Metadata." This analogy makes the value proposition clear without requiring the user to understand the underlying technology. The LLM weights (~2.5 GB) cannot be bundled, so this download is mandatory for users who wish to use the feature.

Based on the emphasis on structured JSON generation, the recommended stack is **Phi-3.5 Mini 3.8B (Q4_K_M quantization)**. Its strong reasoning and instruction-following capabilities make it the most likely to produce perfectly formatted OHMS metadata. The bundle delta is ~2.5 GB as an optional download. The integration sketch involves a lazy-loaded `QThread` worker that instantiates `llama_cpp.Llama`, processes the transcript using a sliding window strategy, streams the GBNF-constrained JSON output back to the main thread, and properly unloads the model upon completion.

## Synthesis and Architectural Recommendation

This analysis has evaluated the technical feasibility of integrating Named Entity Recognition, audio-signal non-verbal event detection, and a local LLM for metadata generation into the Lore application. Each feature was assessed against a strict set of architectural constraints, including a CPU-only environment, no PyTorch dependencies, a sub-400 MB bundle limit, and defined RAM budgets. The findings indicate that while all three features are technically achievable, their simultaneous implementation would violate the project's hard constraints. Therefore, a strategic, phased approach is required to maximize value while managing complexity and resource consumption.

First, the feature priority ranking, based on the highest value-to-complexity ratio for the target user, is as follows:
1.  **Audio-Signal Non-Verbal Event Detection:** This feature delivers the highest value with the least complexity. It begins with a zero-cost implementation using existing VAD outputs for `[Silence]` detection. The addition of a small YAMNet model for `[Laughter]` and `[Crying]` is a justified extension that adds significant value with minimal resource impact.
2.  **Named Entity Recognition (NER):** This feature provides deep semantic enrichment of the transcript, directly supporting the core archival workflow. Its complexity is manageable by positioning it as an optional download, which aligns with the user's stated preference for accuracy over bundling.
3.  **Local LLM Integration for Metadata Generation:** This is the most complex and resource-intensive feature. It requires careful management of large model downloads, long-running background processes, and sophisticated backend logic. While it offers immense value by automating a tedious part of the archival workflow, it carries the highest implementation burden and UX friction.

Second, a cumulative overhead estimate reveals a critical constraint violation. Implementing all three features together would result in a total PyInstaller bundle size of approximately 380 MB (baseline + audio classifier), which is acceptable. However, the peak concurrent RAM usage during transcription would be approximately 4.6 GB (Whisper + NER + Audio Classifier), severely exceeding the 2.7 GB concurrent headroom. The peak RAM usage during abstract generation would be around 4.5 GB (LLM), which is within the 5.7 GB ceiling. This confirms that all three features cannot run simultaneously. The solution is to defer the LLM until a later phase or find ways to reduce the RAM footprint of the other two features, such as by making the NER model truly optional (not just downloadable) so it is never loaded during transcription.

Third, based on the dependency that Phase 4's speaker diarization will provide `speaker_label` data, the following phased placement is recommended:
*   **Phase 5: Audio-Signal Non-Verbal Event Detection:** This feature can be developed and shipped relatively independently. It relies only on the existing audio pipeline and VAD output.
*   **Phase 6: Named Entity Recognition:** This feature can be built next. It will benefit from the `speaker_label` data from Phase 4, allowing for speaker-attributed entity tagging. It should be designed as an optional download.
*   **Phase 7+: Local LLM Integration:** Given its resource intensity and complexity, this feature should be deferred. It would leverage the enriched transcript data from Phases 5 and 6 for potentially better metadata generation. The significant download size necessitates a dedicated UI flow for management.

Fourth, the unified pipeline architecture can be supported by Lore's existing `QThread`/signal/slot pattern. A parent orchestration worker could manage three sibling workers: `TranscriptionWorker`, `AudioClassifierWorker`, and `NERWorker`. All would operate on a shared, thread-safe audio/text buffer. Results from all three would be streamed back to the main thread via signals, where they would be merged and used to populate the `QTextBlockUserData`. The LLM would be a separate, lazy-loaded `QThread` triggered by a distinct UI action, ensuring it does not interfere with the transcription or NER workflows.

Finally, the existing `platformdirs` model cache is well-designed and should be extended with a clear directory structure to manage the new models:

```
models/
├── whisper/
│   └── (Whisper models)
├── nlp/
│   └── (NER model files, if not bundled)
├── audio/
│   └── (YAMNet .tflite model)
└── llm/
    ├── phi-3.5-mini-3b/
    └── llama-3.2-3b/
```

None of these models should be placed in a shared path accessible to the HOARD sibling application, as their formats and purposes are specific to Lore's unique workflows. This structured approach ensures a scalable and maintainable solution for managing multiple optional AI models within the Lore application.
